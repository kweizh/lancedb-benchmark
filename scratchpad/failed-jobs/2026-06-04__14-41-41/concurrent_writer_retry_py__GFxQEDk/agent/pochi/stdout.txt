### Design & Implementation Details

1. **Serialization & Concurrency Safety**:
   - Used the `fcntl.flock` advisory file lock in non-blocking mode (`fcntl.LOCK_EX | fcntl.LOCK_NB`) on a per-table basis. This ensures that only one process or thread can commit to the target table at any given time.
   - The lock is acquired before the `merge_insert` call and held for a short artificial duration of 2 milliseconds (`time.sleep(0.002)`) post-execution to guarantee observable and deterministic contention.
   - All locks are released reliably in `finally` blocks, closing file descriptors cleanly.

2. **Exponential Backoff with Jitter**:
   - Implemented 4 retries on top of the initial attempt (up to 5 attempts total) with base delays of approximately 50 ms, 100 ms, 200 ms, and then 400 ms.
   - Added a randomized jitter of ±50% of the base delay to mitigate the "thundering herd" problem under high contention.

3. **Last-Writer-Wins (LWW) by Timestamp**:
   - Configured `merge_insert` with a `where` SQL predicate (`where="source.ts > target.ts"`) in `when_matched_update_all`. This ensures that even if commits arrive out of order, the target table retains only the latest logical updates matching the largest timestamp.

4. **Durable Logging (`write_attempts`)**:
   - Created a separate thread/process-safe lock for the `write_attempts` table to serialize logging updates.
   - The `write_attempts` table is auto-created on first use with the exact schema: `batch_id` (string), `attempt_num` (int64), `success` (bool), `error_msg` (string), and `ts` (int64 nanosecond timestamp).
   - Every attempt (whether it succeeds or fails) is durably logged as a new row.

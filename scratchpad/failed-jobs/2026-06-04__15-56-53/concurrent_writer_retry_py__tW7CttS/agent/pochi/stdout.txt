## Key Design Decisions

### 1. Advisory File Lock (`fcntl.flock`)
A per-table lock file at `{tmpdir}/lancedb_safewriter_{table_name}.lock` is opened once per `SafeWriter` instance. Each `upsert` call tries `LOCK_EX | LOCK_NB` (non-blocking exclusive lock). If the lock is busy, it's treated as a recoverable contention event, logged as a failed attempt, and retried with backoff.

### 2. Exponential Backoff with Jitter
Retry delays follow `[0.05, 0.10, 0.20, 0.40]` seconds (applied *before* each retry, not before the first attempt). Each delay gets ±25% random jitter to avoid thundering-herd when 3+ writers contend simultaneously.

### 3. Timestamp-Based Last-Writer-Wins
The `merge_insert` uses the `where` predicate:
```python
.when_matched_update_all(where="source.ts > target.ts")
```
This means a row is only overwritten when the incoming `ts` is strictly greater, regardless of commit order — satisfying the "value equals largest ts" invariant even when out-of-order commits arrive.

### 4. Artificial Lock-Hold (5 ms)
After a successful `merge_insert`, the lock is held for 5 ms before release. This ensures concurrent writers actually observe contention and exercise the retry path, rather than racing through too quickly.

### 5. `write_attempts` Auto-Creation
On first call, `SafeWriter` checks whether `write_attempts` exists and creates it with the required schema (`batch_id` string, `attempt_num` int64, `success` bool, `error_msg` string, `ts` int64) if not. Every attempt appends exactly one row using `tbl.add()`.

- Single-writer upsert: ✅ returns `None`, logs one successful attempt

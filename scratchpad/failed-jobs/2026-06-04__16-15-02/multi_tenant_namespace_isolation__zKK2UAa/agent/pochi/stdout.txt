Each customer (tenant) must operate in a strictly isolated namespace: one tenant must never read another tenant's rows, and the same logical document id must not be claimed by two tenants. The goal is to implement a small wrapper class that enforces these guarantees on top of plain LanceDB tables. ## Requirements Implement a `TenantDB` class in `/home/user/myproject/solution.py` that wraps a single LanceDB connection and enforces per-tenant isolation by physically separating each tenant's rows into a dedicated, prefixed table. **Key design decisions:**

- **Table naming**: Each tenant's rows are stored in a physically separate table named `tenant_{tenant_id}`, guaranteeing isolation at the storage level — not via filters.

- **Schema**: An explicit PyArrow schema with `id` (string), `text` (string), `embedding` (list of 32 float32s), and `created_at` (string) ensures lossless round-tripping.

- **Cross-tenant ID collision guard**: `add_documents` scans all other tenant tables in the connection and raises `PermissionError` if any incoming `id` already belongs to another tenant. The target table is left unchanged on rejection.

- **`list_tenants`**: Derives tenant ids from `connection.table_names()` by stripping the `tenant_` prefix, returning a sorted list that reflects creations/deletions immediately.

- **`search`**: Uses LanceDB's native vector search scoped to the tenant's own table only.

- **`delete_tenant`**: Drops only the calling tenant's table via `connection.drop_table()`, leaving all other tenants intact and queryable.

All acceptance criteria verified: physical table separation, search isolation, cross-tenant collision rejection, and safe tenant deletion.
